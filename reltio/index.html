Core OOP Pillars
Q What are the 4 pillars of OOP? Explain each like I'm new to programming. ▾
Easy
Answer
The 4 pillars are Encapsulation, Abstraction, Inheritance, and Polymorphism.
Encapsulation — Think of a TV remote. You press buttons (public methods), but you don't see the circuit board inside (private data). The internals are hidden. In code: private fields + public getters/setters.

Abstraction — When you drive a car, you use the steering wheel. You don't think about how the engine combustion works. Abstraction = showing only what's needed, hiding the how. In code: interfaces and abstract classes.

Inheritance — A "SportsCar" is a type of "Car". It inherits all car properties (has wheels, has engine) but adds its own (turbo, spoiler). In Java: class SportsCar extends Car.

Polymorphism — The same word "speak" means different things for a Dog (bark), Cat (meow), and Human (talk). Same method name, different behaviour based on the object. Runtime polymorphism = method overriding.

Q What happens if you override equals() but NOT hashCode()? ▾
Hard — VERY common at Reltio
Your code will break silently when using HashMaps. Here's exactly why:
Imagine a library where books are shelved by first letter of title. HashMap works the same way — it uses hashCode() to find which "shelf" (bucket) to look in, then equals() to find the exact book.

If you say two books are "equal" (same content) but give them different hashCodes, HashMap stores them on different shelves. When you go looking for one, it checks the wrong shelf and says "not found!" — even though the book exists.

Rule: If A.equals(B) is true → A.hashCode() must equal B.hashCode(). ALWAYS override both together.
JAVA
// WRONG — only overrides equals
class Student {
    int id; String name;

    public boolean equals(Object o) {
        return ((Student)o).id == this.id; // logically same student
    }
    // hashCode not overridden — still uses memory address!
}
// HashMap will FAIL to find the student even if it was inserted.

// CORRECT — override both
public int hashCode() { return Objects.hash(id); }
public boolean equals(Object o) {
    if (this == o) return true;
    if (!(o instanceof Student)) return false;
    return this.id == ((Student)o).id;
}
Q Explain == vs .equals() and the String Pool. ▾
Medium
== asks: "Are these the SAME house (same memory address)?"
.equals() asks: "Do these houses LOOK THE SAME (same content)?"

String Pool is Java's trick to save memory. When you write String s = "hello", Java checks if "hello" already exists in its pool. If yes, it reuses it. So two string literals with same value share the same object — making == return true by accident. But new String("hello") always creates a NEW house outside the pool.
JAVA
String a = "hello";
String b = "hello";
String c = new String("hello");

a == b        // TRUE  — both point to same pool object
a == c        // FALSE — c is a new object in heap
a.equals(c)  // TRUE  — same content "hello"

// Tricky Integer cache (-128 to 127)
Integer x = 127; Integer y = 127; // x == y → TRUE (cached)
Integer p = 200; Integer q = 200; // p == q → FALSE (not cached)
Q Write a thread-safe Singleton using Double-Checked Locking. ▾
Hard
Singleton = Only ONE instance of a class should EVER exist. Like a country's President — there's only one.

The problem: what if 2 threads both check "does the president exist?" at the same moment? Both say no → both create one → now you have TWO presidents. That's the race condition we're solving.

Double-checked locking = First check without locking (fast), then lock and check again (safe). volatile prevents the CPU from reordering instructions which could cause a half-built object to be returned.
JAVA
public class Singleton {
    // volatile = changes visible to ALL threads immediately
    private static volatile Singleton instance;

    private Singleton() {} // private constructor

    public static Singleton getInstance() {
        if (instance == null) {          // 1st check (no lock — fast)
            synchronized(Singleton.class) {
                if (instance == null) {   // 2nd check (inside lock — safe)
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}
Reltio cares about design patterns. Also know Factory (creates objects without exposing creation logic) and Observer (one object notifies many when it changes).
Q How does HashMap work internally? What's a hash collision? ▾
Medium
HashMap is like a parking lot with numbered slots. When you park (put a key), Java runs hashCode() on your key to get a slot number. When you want your car back (get), it runs hashCode() again, goes to that slot, and checks if the car is yours using equals().

Collision = Two different keys get the same slot number. Java handles this by chaining — each slot is a linked list of entries. Java 8+ converts this list to a TreeMap when it gets more than 8 entries (so lookup stays O(log n) not O(n)).
Slot 0: [ ] Slot 1: [ "apple"→5 ] → [ "grape"→8 ] ← collision! same hash Slot 2: [ "banana"→3 ] Slot 3: [ ] ... Slot N: [ "mango"→1 ] Time: O(1) average for get/put Worst case (all collisions): O(n) → that's why hashCode() matters!
Common follow-up: "What's the default load factor?" 0.75 — when HashMap is 75% full, it resizes to double (rehashing). This is why initial capacity matters for performance.
Q throw vs throws, checked vs unchecked exceptions ▾
Easy
throw = Actually hurling the exception right now in your code.
throws = Warning on a method saying "Hey, I might throw this — caller, you deal with it."

Checked = Problems Java FORCES you to handle (file not found, network error). The compiler won't let you ignore these. Examples: IOException, SQLException.

JAVA
// throws = declaration (the warning sign)
public void readFile(String path) throws IOException {
    // throw = actually throwing it
    if (path == null) throw new IllegalArgumentException("No path");
    // ... file reading code
}
// IOException is Checked — caller MUST handle or re-declare it
// IllegalArgumentException is Unchecked — no forced handling
Q Explain Java memory: Stack vs Heap. What causes StackOverflowError? ▾
Medium
Stack = A stack of plates. Every method call adds a plate (frame). Local variables live in that plate. When the method returns, the plate is removed. Each thread has its own stack. If you keep calling methods forever (infinite recursion), the stack overflows — all the plates fall. → StackOverflowError.

Heap = A big warehouse. ALL objects live here. Threads share this warehouse. Garbage Collector cleans up objects nobody is pointing to anymore. When warehouse is full → OutOfMemoryError.
THREAD 1 STACK SHARED HEAP ┌──────────────┐ ┌──────────────────────────┐ │ main() │ │ Young Gen (new objects) │ │ int x = 5 │──────▶│ String obj "hello" │ │ obj ref ──────────▶│ MyClass instance │ ├──────────────┤ │ │ │ methodA() │ │ Old Gen (surviving objs) │ │ int y = 10 │ └──────────────────────────┘ └──────────────┘ Stack = local vars + refs Heap = actual objects

Q Comparable vs Comparator — what's the difference? ▾
Medium
Comparable = The object knows how to compare itself. Like a student who knows their own rank: "I'm better than you because my GPA is higher." You implement this inside the class.

Comparator = An external judge who compares two objects by whatever rule you want. Today sort by name, tomorrow sort by salary — you write different Comparators without touching the class.
JAVA
// Comparable — implemented inside the class
class Student implements Comparable<Student> {
    int gpa;
    public int compareTo(Student other) {
        return this.gpa - other.gpa; // natural order = ascending GPA
    }
}
Collections.sort(students); // uses compareTo automatically

// Comparator — external, flexible
Comparator<Student> byName = (a, b) -> a.name.compareTo(b.name);
Collections.sort(students, byName); // sort by name instead

Q Vertical Order Traversal of a Binary Tree — ACTUAL Reltio Hard Question ▾
Hard — Asked in Reltio Campus Drive
Imagine shining a light straight down on your binary tree from above. Each node casts a shadow on a number line. Root is at position 0. Going left = -1. Going right = +1. Collect all nodes that land on the same vertical line, then print each line from left to right.
1 col: 0 / \ 2 3 2 → col:-1, 3 → col:+1 / \ \ 4 5 6 4→col:-2, 5→col:0, 6→col:+2 Vertical Output: [4], [2], [1,5], [3], [6] (nodes at same col sorted by row, then value)
Algorithm: BFS with a queue storing (node, row, col). Root = (0,0). Left child = col-1, Right child = col+1.
JAVA
import java.util.*;

class Solution {

    // Helper class to store node info in the queue
    static class NodeInfo {
        TreeNode node;
        int row, col;
        NodeInfo(TreeNode n, int r, int c) {
            node = n; row = r; col = c;
        }
    }

    public List<List<Integer>> verticalTraversal(TreeNode root) {
        List<List<Integer>> result = new ArrayList<>();
        if (root == null) return result;

        // col → list of (row, val) pairs
        TreeMap<Integer, List<int[]>> colMap = new TreeMap<>();
        Queue<NodeInfo> queue = new LinkedList<>();
        queue.offer(new NodeInfo(root, 0, 0));

        while (!queue.isEmpty()) {
            NodeInfo curr = queue.poll();
            colMap.computeIfAbsent(curr.col, k -> new ArrayList<>())
                  .add(new int[]{curr.row, curr.node.val});

            if (curr.node.left  != null)
                queue.offer(new NodeInfo(curr.node.left,  curr.row+1, curr.col-1));
            if (curr.node.right != null)
                queue.offer(new NodeInfo(curr.node.right, curr.row+1, curr.col+1));
        }

        // TreeMap already sorted by col; sort each column by row then value
        for (List<int[]> entries : colMap.values()) {
            entries.sort((a, b) -> a[0] != b[0] ? a[0]-b[0] : a[1]-b[1]);
            List<Integer> colVals = new ArrayList<>();
            for (int[] e : entries) colVals.add(e[1]);
            result.add(colVals);
        }
        return result;
    }
}
// Time: O(N log N)  |  Space: O(N)
Time
O(N log N)
Space
O(N)
Q Lowest Common Ancestor (LCA) of a Binary Tree ▾
Medium
LCA = The first "fork in the road" where two paths diverge.
You and your cousin share the same grandparent — that's your LCA. The grandparent is the deepest ancestor you both share.
JAVA
class Solution {

    public TreeNode lowestCommonAncestor(TreeNode root, TreeNode p, TreeNode q) {
        // Base: reached null OR found p or q — return whatever we have
        if (root == null || root == p || root == q)
            return root;

        TreeNode left  = lowestCommonAncestor(root.left,  p, q);
        TreeNode right = lowestCommonAncestor(root.right, p, q);

        // p found on left, q found on right → this node is LCA
        if (left != null && right != null)
            return root;

        // Only one side found something — propagate it upward
        return (left != null) ? left : right;
    }
}
// Time: O(N)  |  Space: O(H) where H = height of tree
// BST shortcut: p,q both < root → go left
//               p,q both > root → go right
//               else → root IS the LCA
Q Serialize and Deserialize a Binary Tree ▾
Hard
Serialize = Take your tree and convert it to a flat string that can be saved to disk or sent over a network. Like writing a family tree on paper.
Deserialize = Read that string and rebuild the tree exactly. Like reconstructing the family tree from your notes.
JAVA
import java.util.*;

class Codec {

    // Encode tree to a single string using BFS level-order
    public String serialize(TreeNode root) {
        if (root == null) return "";
        StringBuilder sb = new StringBuilder();
        Queue<TreeNode> q = new LinkedList<>();
        q.offer(root);

        while (!q.isEmpty()) {
            TreeNode node = q.poll();
            if (node == null) {
                sb.append("N,");          // N marks a null child
            } else {
                sb.append(node.val).append(",");
                q.offer(node.left);        // add children (even if null)
                q.offer(node.right);
            }
        }
        return sb.toString();
        // Tree 1→2→3 serializes to "1,2,3,N,N,N,N,"
    }

    // Decode string back into tree
    public TreeNode deserialize(String data) {
        if (data.isEmpty()) return null;
        String[] vals = data.split(",");
        TreeNode root = new TreeNode(Integer.parseInt(vals[0]));
        Queue<TreeNode> q = new LinkedList<>();
        q.offer(root);
        int i = 1;

        while (!q.isEmpty() && i < vals.length) {
            TreeNode node = q.poll();
            if (!vals[i].equals("N")) {
                node.left = new TreeNode(Integer.parseInt(vals[i]));
                q.offer(node.left);
            }
            i++;
            if (i < vals.length && !vals[i].equals("N")) {
                node.right = new TreeNode(Integer.parseInt(vals[i]));
                q.offer(node.right);
            }
            i++;
        }
        return root;
    }
}

// TreeNode definition (standard for all tree problems):
// class TreeNode {
//     int val;
//     TreeNode left, right;
//     TreeNode(int val) { this.val = val; }
// }

Q Climbing Stairs / 100 Staircase — ACTUAL Reltio Question ▾
Medium — Asked in Reltio Campus Drive
To reach step N, you either came from step N-1 (took 1 step) or from step N-2 (took 2 steps). So ways(N) = ways(N-1) + ways(N-2). This is exactly Fibonacci! Base cases: 1 way to reach step 1, 2 ways to reach step 2 (1+1 or 2).
Steps: 1→1 2→2 3→3 4→5 5→8 6→13 Pattern: ways(n) = ways(n-1) + ways(n-2) Trace for n=4: ways(1)=1, ways(2)=2 ways(3)=ways(2)+ways(1)=3 ways(4)=ways(3)+ways(2)=5 ✓
JAVA
class Solution {

    // Optimal: O(n) time, O(1) space — rolling two variables
    public int climbStairs(int n) {
        if (n <= 2) return n;
        int a = 1, b = 2;             // a = ways(i-2), b = ways(i-1)
        for (int i = 3; i <= n; i++) {
            int temp = a + b;          // ways(i) = ways(i-1) + ways(i-2)
            a = b;
            b = temp;
        }
        return b;
    }

    // Variant: can climb 1, 2, OR 3 steps at a time
    public int climbStairs3(int n) {
        int[] dp = new int[n + 1];
        dp[0] = 1; dp[1] = 1; dp[2] = 2;
        for (int i = 3; i <= n; i++) {
            dp[i] = dp[i-1] + dp[i-2] + dp[i-3];
        }
        return dp[n];
    }

    // Example: n=4 → output 5
    // (1+1+1+1), (1+1+2), (1+2+1), (2+1+1), (2+2)
}
Reltio asked candidates to explain the recurrence relation. Say: "The number of ways to reach step N equals ways to reach N-1 plus ways to reach N-2 — because those are the only two steps you could have come from."
Q 0/1 Knapsack — fit the most valuable items in a bag ▾
Hard
You're packing for a trip with a bag that holds W kg. Each item has a weight and a value. You want maximum value without exceeding W. For each item, you have exactly 2 choices: take it or skip it (that's why it's "0/1").
JAVA
class Solution {

    // Space-optimised 1D DP version — O(n*W) time, O(W) space
    public int knapsack(int[] weights, int[] values, int W) {
        int n = weights.length;
        int[] dp = new int[W + 1];  // dp[w] = max value with capacity w

        for (int i = 0; i < n; i++) {
            // ⚠️  Traverse RIGHT TO LEFT to avoid picking item i twice!
            for (int w = W; w >= weights[i]; w--) {
                // Skip item i   vs   take item i
                dp[w] = Math.max(dp[w], values[i] + dp[w - weights[i]]);
            }
        }
        return dp[W];
    }

    // Example usage:
    // int[] weights = {2, 3, 4};
    // int[] values  = {3, 4, 5};
    // int W = 5;
    // knapsack(weights, values, W) → 7
    // (take item0 weight=2,val=3 + item1 weight=3,val=4 = total val 7)
}
Time
O(n × W)
Space
O(W)
Unbounded Knapsack (Coin Change) — can reuse items → traverse LEFT to RIGHT instead of right to left.
Q Longest Increasing Subsequence (LIS) — O(n log n) solution ▾
Hard
Find the longest chain of numbers that keep going up. From [10,9,2,5,3,7,101,18], the answer is [2,3,7,101] — length 4. Numbers don't have to be adjacent, just in increasing order left to right.

O(n log n) trick: maintain a "tails" array where tails[i] = smallest possible tail of any increasing subsequence of length i+1. Use binary search to update it. The length of tails at the end is your answer.
JAVA
class Solution {

    // O(n log n) solution using binary search + patience sort
    public int lengthOfLIS(int[] nums) {
        List<Integer> tails = new ArrayList<>();
        // tails[i] = smallest possible tail for LIS of length i+1

        for (int num : nums) {
            int pos = binarySearch(tails, num);
            if (pos == tails.size()) {
                tails.add(num);          // num extends the longest LIS
            } else {
                tails.set(pos, num);     // replace to keep tail as small as possible
            }
        }
        return tails.size();
    }

    // Find leftmost position where tails[pos] >= num (lower bound)
    private int binarySearch(List<Integer> tails, int num) {
        int lo = 0, hi = tails.size();
        while (lo < hi) {
            int mid = lo + (hi - lo) / 2;
            if (tails.get(mid) < num) lo = mid + 1;
            else                       hi = mid;
        }
        return lo;
    }

    // Trace: [10,9,2,5,3,7,101,18]
    // 10→[10]  9→[9]  2→[2]  5→[2,5]  3→[2,3]
    // 7→[2,3,7]  101→[2,3,7,101]  18→[2,3,7,18]  → answer = 4
}
The tails array does NOT represent an actual LIS — it's a tool for counting. Mentioning O(n log n) and explaining why will genuinely impress the interviewer.
Q Longest Substring Without Repeating Characters ▾
Medium
Imagine a sliding window on a conveyor belt of characters. Keep expanding the right side. Whenever a duplicate enters the window, shrink from the left until the duplicate is gone. Track the max window size you ever had.
JAVA
class Solution {

    public int lengthOfLongestSubstring(String s) {
        Map<Character, Integer> lastSeen = new HashMap<>(); // char → last index
        int left = 0, maxLen = 0;

        for (int right = 0; right < s.length(); right++) {
            char ch = s.charAt(right);

            // If char seen before AND it's inside our current window
            if (lastSeen.containsKey(ch) && lastSeen.get(ch) >= left) {
                left = lastSeen.get(ch) + 1;   // shrink window from left
            }
            lastSeen.put(ch, right);
            maxLen = Math.max(maxLen, right - left + 1);
        }
        return maxLen;
    }

    // Example: "abcabcbb" → 3 ("abc")
    // Example: "pwwkew"  → 3 ("wke")
    // Time: O(n)  |  Space: O(min(n, charset))
}
Q LRU Cache — O(1) get and put ▾
Hard — Tests DS Design
LRU = Least Recently Used. Like your phone's recent apps list. When it's full and you open a new app, the one you used longest ago gets kicked out. You need: fast lookup (HashMap) + ordering by recency (Doubly Linked List).
JAVA
import java.util.*;

// Java's LinkedHashMap already gives us HashMap + insertion order = perfect LRU!
class LRUCache extends LinkedHashMap<Integer, Integer> {
    private int capacity;

    public LRUCache(int capacity) {
        // accessOrder=true means: on get/put, move entry to END (most recent)
        super(capacity, 0.75f, true);
        this.capacity = capacity;
    }

    public int get(int key) {
        return super.getOrDefault(key, -1);  // auto-moves to end
    }

    public void put(int key, int value) {
        super.put(key, value);                 // auto-moves to end
    }

    // Called automatically when a new entry is added — evict oldest if over capacity
    @Override
    protected boolean removeEldestEntry(Map.Entry<Integer, Integer> eldest) {
        return size() > capacity;
    }
}

// ─── Manual approach (interviewer may ask for this) ───────────────────────
// HashMap: key → DoublyLinkedList Node
// DLL:     head (oldest/LRU) ↔ ... ↔ tail (newest/MRU)
// get(key):  find node via map, move to tail, return val   → O(1)
// put(key):  add node at tail, if over capacity remove head → O(1)
Q Detect Cycle in a Directed Graph ▾
Medium
Walking through a city. You're exploring streets (DFS). If you end up back on a street you're CURRENTLY walking on (in your current path, not just any visited street), you've found a cycle. Track your current path with a "recursion stack."
JAVA
import java.util.*;

class Solution {
    private List<List<Integer>> graph;
    private boolean[] visited;
    private boolean[] recStack;   // nodes in CURRENT DFS path

    public boolean hasCycle(int V, List<List<Integer>> adj) {
        graph    = adj;
        visited  = new boolean[V];
        recStack = new boolean[V];

        for (int v = 0; v < V; v++) {
            if (!visited[v] && dfs(v))
                return true;
        }
        return false;
    }

    private boolean dfs(int node) {
        visited[node]  = true;
        recStack[node] = true;     // mark as part of current path

        for (int neighbor : graph.get(node)) {
            if (!visited[neighbor]) {
                if (dfs(neighbor)) return true;
            } else if (recStack[neighbor]) {
                return true;            // back edge found → CYCLE!
            }
        }
        recStack[node] = false;    // leaving this path, unmark
        return false;
    }
}
// Time: O(V+E)  |  Space: O(V)
Q Topological Sort — Kahn's Algorithm (BFS) ▾
Medium
Like scheduling tasks where some must happen before others. You can't submit a PR before writing code. You can't deploy before tests pass. Topological sort gives a valid order. Start with tasks that have zero dependencies — process them, remove their edges, find new zero-dependency tasks, repeat.
JAVA
import java.util.*;

class Solution {

    public List<Integer> topoSort(int V, int[][] edges) {
        int[] inDegree = new int[V];
        List<List<Integer>> graph = new ArrayList<>();
        for (int i = 0; i < V; i++) graph.add(new ArrayList<>());

        // Build adjacency list and count incoming edges per node
        for (int[] edge : edges) {
            graph.get(edge[0]).add(edge[1]);
            inDegree[edge[1]]++;
        }

        // Enqueue all nodes with zero incoming edges (no dependencies)
        Queue<Integer> q = new LinkedList<>();
        for (int i = 0; i < V; i++)
            if (inDegree[i] == 0) q.offer(i);

        List<Integer> order = new ArrayList<>();

        while (!q.isEmpty()) {
            int node = q.poll();
            order.add(node);
            for (int nei : graph.get(node)) {
                inDegree[nei]--;                    // remove this edge
                if (inDegree[nei] == 0) q.offer(nei); // now ready!
            }
        }

        // If result has fewer nodes than V → cycle exists, topo sort impossible
        return order.size() == V ? order : new ArrayList<>();
    }
}
// Time: O(V+E)  |  Space: O(V+E)
// Use case: course prerequisites, build systems, task scheduling

Core Concepts Quick Reference
ConceptPlain English Explanation
ACIDAtomicity (all or nothing — bank transfer either fully succeeds or fully fails), Consistency (data always valid — you can't have -$500 balance), Isolation (two users buying the last seat don't interfere), Durability (committed data survives crashes)
IndexingLike a book's index page. Instead of reading every page to find "Knapsack", you jump to the index, find the page number, go directly there. B-Tree index = O(log n) lookup instead of O(n) full table scan.
Clustered IndexThe data is physically SORTED by this column on disk. Like a dictionary — words are in alphabetical order. Only ONE per table.
Non-clustered IndexA separate lookup table with pointers to actual rows. Like a book's index pointing to page numbers. Multiple allowed per table.
NormalizationOrganizing data to reduce repetition. 1NF = one value per cell. 2NF = no partial dependency. 3NF = no transitive dependency (A→B→C should become two tables).
WHERE vs HAVINGWHERE filters BEFORE grouping (filters individual rows). HAVING filters AFTER grouping (filters groups). HAVING works on aggregate functions like COUNT(), SUM().

Q Window Functions — ROW_NUMBER, RANK, DENSE_RANK, LAG, LEAD ▾
Hard — Often missing from prep guides
Window functions look at a "window" of rows around each row — without collapsing them like GROUP BY does. You can rank rows, compare to previous rows, calculate running totals — all while keeping every row visible.
SQL
-- ROW_NUMBER: unique sequential number (no ties)
-- RANK: same rank for ties, skips next number
-- DENSE_RANK: same rank for ties, NO gaps
SELECT name, salary,
    ROW_NUMBER()  OVER (ORDER BY salary DESC) AS row_num,
    RANK()        OVER (ORDER BY salary DESC) AS rnk,
    DENSE_RANK()  OVER (ORDER BY salary DESC) AS d_rnk
FROM employees;

-- LAG/LEAD: look at previous/next row
-- Compare each employee's salary to previous one
SELECT name, salary,
    LAG(salary, 1) OVER (ORDER BY hire_date) AS prev_salary,
    salary - LAG(salary,1) OVER (ORDER BY hire_date) AS change
FROM employees;

-- PARTITION BY: apply window per department
SELECT dept, name, salary,
    RANK() OVER (PARTITION BY dept ORDER BY salary DESC) AS dept_rank
FROM employees;

Processes, memory, scheduling, and the famous "what happens when you type a URL?" question.
Core Concepts
ConceptPlain English
Process vs ThreadProcess = a program running in isolation with its own memory (like separate apartments). Thread = multiple workers inside one apartment, sharing the same kitchen (heap) but each with their own room (stack).
Context SwitchCPU saves current task's state (like bookmarking a book page) and picks up another task from where IT left off. Overhead: saving/restoring registers, cache invalidation.
DeadlockThread A holds your car key and wants your house key. Thread B holds your house key and wants your car key. Both wait forever. 4 conditions must ALL hold simultaneously to cause deadlock.
PagingRAM is divided into equal-size "pages." Each process thinks it has contiguous memory but pages are scattered. Page table maps virtual → physical. No external fragmentation.
Virtual MemoryYour program thinks it has way more RAM than physically exists. Pages not currently needed are parked on disk. When accessed → page fault → OS loads it from disk.
ThrashingWhen the system spends more time swapping pages in/out than actually running programs. RAM is too full. CPU usage drops to near zero — everything grinds to halt.
Semaphore vs MutexMutex = bathroom key (only ONE person, must be SAME person who took it). Semaphore = parking lot with N spaces, any car can take/free a spot.
Key Q&A
Q What are the 4 deadlock conditions? How do you prevent deadlock? ▾
Hard
ALL 4 must be true simultaneously for a deadlock to occur. Remove ANY ONE and deadlock cannot happen.
1. Mutual Exclusion — A resource can only be held by one thread at a time (e.g., a printer).
2. Hold & Wait — A thread holds one resource while waiting for another.
3. No Preemption — Resources cannot be forcibly taken away; a thread must release voluntarily.
4. Circular Wait — Thread A waits for B, B waits for C, C waits for A — a circle of waiting.
Prevention strategies:
• Break Hold & Wait: require threads to request ALL resources at once before starting.
• Break Circular Wait: always acquire locks in a fixed global order (always lock mutex1 THEN mutex2, never the reverse).
• Break No Preemption: use tryLock() with timeout — if you can't get the second lock, release the first and try again.
Q What happens when you type a URL in a browser? (Full deep answer) ▾
Hard — OS + Networking combined
  1. DNS Resolution — Browser checks its own cache → OS cache → ISP DNS server → Root DNS → TLD DNS → Authoritative DNS. Result: IP address for the domain.
  2. TCP 3-Way Handshake — Client sends SYN. Server replies SYN-ACK. Client sends ACK. Connection established. (Think: "Hey" → "Hey back, ready?" → "OK let's go")
  3. TLS Handshake (HTTPS) — Client and server negotiate encryption, exchange certificates, verify identity, agree on session key. All future data is encrypted.
  4. HTTP Request — Browser sends GET /path HTTP/1.1 with headers (cookies, Accept-Language, etc.)
  5. Server Processing — OS receives packet, TCP stack delivers to correct port (80/443), web server process handles it. May query database, call microservices, read files.
  6. HTTP Response — Server sends back 200 OK with HTML content (or 301 redirect, 404 etc.)
  7. Browser Rendering — Parses HTML → builds DOM. Fetches CSS, JS, images (each potentially a new TCP connection or reusing HTTP/2 multiplexed). Renders page visually.
This question tests your depth. Each step could be a full conversation. Show breadth by naming all steps, then go deep on whichever one the interviewer probes.

Q Two threads print odd and even numbers alternately — ACTUAL Reltio Question ▾
Hard — Directly from Reltio Campus Drive
Think of it as two relay runners sharing a baton. ODD runner prints an odd number, passes the baton (notifyAll), then waits. EVEN runner wakes up, prints an even number, passes the baton back, waits. The shared "baton" state is a boolean flag called isOddTurn.
JAVA — Full Solution
class OddEvenPrinter {
    private int num = 1;
    private final int max;
    private boolean isOddTurn = true;  // ODD goes first

    OddEvenPrinter(int max) { this.max = max; }

    synchronized void printOdd() throws InterruptedException {
        while (num <= max) {
            while (!isOddTurn) wait();      // not my turn? wait
            if (num <= max)
                System.out.println("ODD:  " + num++);
            isOddTurn = false;             // give turn to EVEN
            notifyAll();                    // wake up EVEN thread
        }
    }

    synchronized void printEven() throws InterruptedException {
        while (num <= max) {
            while (isOddTurn) wait();       // not my turn? wait
            if (num <= max)
                System.out.println("EVEN: " + num++);
            isOddTurn = true;              // give turn back to ODD
            notifyAll();
        }
    }
}

public class Main {
    public static void main(String[] args) {
        OddEvenPrinter printer = new OddEvenPrinter(10);

        Thread t1 = new Thread(() -> {
            try { printer.printOdd(); }
            catch (InterruptedException e) {}
        });
        Thread t2 = new Thread(() -> {
            try { printer.printEven(); }
            catch (InterruptedException e) {}
        });

        t1.start(); t2.start();
    }
}
// Output: ODD: 1  EVEN: 2  ODD: 3  EVEN: 4 ... EVEN: 10
Key points the interviewer checks: (1) synchronized on the shared object, (2) while loop around wait() — not if, (3) notifyAll() not notify() to avoid missing wake-ups, (4) the flag correctly alternates.
Q Write code that causes a deadlock. Then fix it. ▾
Hard
JAVA
Object lock1 = new Object();
Object lock2 = new Object();

// ❌ DEADLOCK: Thread A and B acquire locks in OPPOSITE order
Thread A = new Thread(() -> {
    synchronized(lock1) {
        sleep(50); // give B time to grab lock2
        synchronized(lock2) { /* never reaches here */ }
    }
});
Thread B = new Thread(() -> {
    synchronized(lock2) {      // B holds lock2
        synchronized(lock1) { /* waits for A to release lock1 */ }
    }
});
// A holds lock1, wants lock2. B holds lock2, wants lock1. Deadlock!

// ✅ FIX: Always acquire locks in the SAME order in ALL threads
Thread A_fixed = new Thread(() -> {
    synchronized(lock1) { synchronized(lock2) { /* works */ } }
});
Thread B_fixed = new Thread(() -> {
    synchronized(lock1) { synchronized(lock2) { /* same order */ } }
});
// Both now compete for lock1 first → no circular wait
Q volatile vs synchronized — when is volatile NOT enough? ▾
Medium
volatile = A whiteboard visible to all team members. Everyone reads the latest value. But if two people try to update it at the same moment, they can still collide.

i++ is actually THREE operations: READ i → ADD 1 → WRITE i. volatile makes reads/writes atomic individually, but can't prevent two threads interleaving between the read and write.

For compound operations like i++, use AtomicInteger.incrementAndGet() (hardware-level atomic) or synchronized.
JAVA
volatile int counter = 0;
counter++; // ❌ NOT thread-safe! Not atomic.

AtomicInteger counter = new AtomicInteger(0);
counter.incrementAndGet(); // ✅ Thread-safe, faster than synchronized

Q What happens during a TLS handshake? (HTTPS) ▾
Hard
TLS handshake = Two strangers establishing a secret language before talking publicly.

1. "Here's what languages I speak" (client sends supported ciphers)
2. "Let's use this one. Here's my ID card (certificate)" (server picks cipher, sends cert)
3. "I verified your ID. Let's agree on a secret word" (key exchange)
4. Both derive the same session key from the exchange
5. "Test: can you understand this?" (both send encrypted "Finished" messages)
6. Now all actual data is encrypted with that symmetric session key.
The clever part: they agree on a shared secret (session key) over a public channel without ever sending the key itself — using asymmetric crypto (RSA or ECDHE) for the handshake, then symmetric AES for the actual data (much faster).

A rate limiter says "you can make maximum 100 API calls per minute." Common algorithms:

Token Bucket — Bucket holds N tokens. Each request consumes one token. Tokens refill at a fixed rate. If bucket is empty, request is rejected. Allows bursting (use all tokens at once).

Q What is Master Data Management (MDM) and why is it hard? ▾
Medium
Imagine a large hospital. Patient "John Smith, DOB 1985" exists in 5 different systems: billing has his old address, pharmacy has his new phone, EMR has a typo "Jon Smith", insurance has his maiden name. MDM says: "Let me find all these records, figure out they're all the same John Smith, and create ONE perfect unified record."

That ONE perfect record is called the Golden Record. It's the authoritative truth.
Why it's hard:
• Same entity has different names ("IBM" vs "International Business Machines" vs "I.B.M.")
• Data quality varies across sources — some have typos, some are incomplete
• Changes in real-time — customer moves, changes name, gets married
• Scale — billions of records, O(n²) comparisons is impossible
• Conflicting values — which phone number is "correct"? You need rules (survivorship rules).
• Reltio solves this across multi-cloud enterprise systems in real-time.
Q What is a Golden Record? How is it created? ▾
Medium
Golden Record = The single best version of truth for an entity, assembled by picking the best attribute value from all sources.

For "John Smith": 3 systems disagree on his phone number. You apply survivorship rules:
• "Most recent" → use the latest updated system's value
• "Most trusted source" → CRM data beats legacy warehouse data
• "Most complete" → prefer non-null values

The Golden Record is assembled from these rules and becomes what all downstream systems read.

Q Design a simple entity matching system from scratch ▾
Hard — Shows System Thinking + ML + Reltio Domain
  1. Blocking — Don't compare every record to every other (O(n²) is too slow). Group records that might match: same first letter + same ZIP, or same company name prefix. Reduces comparison space by 99%.
  2. Feature Extraction — For each candidate pair, compute: Jaro-Winkler similarity for names, Soundex code match for phonetics, exact match for IDs, edit distance for addresses.
  3. Classification — Train a binary classifier (logistic regression or gradient boosting) on labeled pairs (match / no-match). Input: similarity features. Output: probability of match.
  4. Clustering — Records above match threshold are connected. Use Union-Find / connected components to group all linked records into one entity cluster.
  5. Survivorship — Pick the best attribute values for the golden record using rules (most recent, most complete, most trusted).
  6. Feedback Loop — Human reviewers label uncertain cases (close to 0.5 probability). These labeled pairs retrain the model. Accuracy improves over time.
Q CAP Theorem — and what does Reltio prioritize? ▾
Hard
In a distributed system (multiple servers), you can only guarantee 2 out of 3:

Consistency — Every server shows the same data at the same time. "Read what you just wrote."
Availability — Every request gets a response (even if it might be stale).
Partition Tolerance — System keeps working even if some servers lose network connection to others.

Network failures are unavoidable in multi-cloud → you MUST have Partition Tolerance. So the real choice is:
• CP: consistent data, might be briefly unavailable during partition
• AP: always responds, but might serve slightly stale data
Enterprise MDM like Reltio typically prioritizes Consistency (CP). A golden record with wrong data (stale) is MORE damaging to a business than a brief unavailability. "I'd rather have no data for 2 seconds than wrong data that drives incorrect decisions."
